# 角色 A 任务计划：词法分析与语法分析闭环

## 1. 目标

围绕 ToyC 编译器前端，完成从源代码字符串到 AST 的完整链路，确保后续语义分析和 IR 生成模块能够直接消费角色 A 的输出。

## 2. 当前工作区进度检查

### 已完成部分

- 已经建立前端核心文件结构，包括 [src/ast.h](../src/ast.h)、[src/lexer.h](../src/lexer.h)、[src/lexer.cpp](../src/lexer.cpp)、[src/parser.h](../src/parser.h)、[src/parser.cpp](../src/parser.cpp)。
- 词法分析器已支持：
  - 标识符、数字、关键字
  - 注释跳过
  - 基本运算符与分隔符
  - 负数字面量的初步识别
- 语法分析器已实现：
  - 递归下降解析框架
  - 表达式的优先级与结合性处理
  - 块语句、if/while、return、break/continue、函数定义、变量声明等基础结构
- 测试骨架已建立，见 [tests/test_lexer.cpp](../tests/test_lexer.cpp) 与 [tests/test_parser.cpp](../tests/test_parser.cpp)。

### 当前缺口

- 词法分析器还需要补齐更严密的错误处理与边界情况，例如非法字符、连续运算符、注释结尾异常。
- 语法分析器需要继续强化对完整 ToyC 语法的覆盖，尤其是：
  - 全局声明与函数定义混合的完整解析
  - 局部声明与块级作用域的边界处理
  - 复杂嵌套语句和表达式的回溯稳定性
- 目前缺少面向调试与验收的 AST 打印/可视化辅助工具，难以直观验证树形结构。
- 测试覆盖仍偏基础，需要补足更多典型样例与异常样例。

## 3. 下一阶段任务拆解

### 任务 1：完成词法分析器收口

- 目标：确保词法分析器稳定识别完整 ToyC 的终结符集。
- 具体事项：
  - 补齐对非法字符的报错策略。
  - 统一 token 的行列号信息。
  - 增加对 `&&`、`||`、`!=`、`==`、`<=`、`>=` 的稳定识别。
  - 增加针对注释和空白的回归测试。

### 任务 2：补全语法分析器

- 目标：把前端从“能跑通简单例子”提升为“可覆盖大多数 ToyC 语法”。
- 具体事项：
  - 进一步验证函数定义、形参列表、复合语句、if/else、while、return、break/continue 的解析正确性。
  - 确保声明语句与表达式语句在块内和块外都能正确被区分。
  - 对语法错误给出明确信息并尽量保持可恢复性。

### 任务 3：补充 AST 调试能力

- 目标：便于角色 B 和后续联调时快速检查 AST 是否符合预期。
- 具体事项：
  - 增加一个简单的 AST 打印函数，用于打印树结构。
  - 在测试中输出解析结果，便于人工检查。

### 任务 4：完善自测用例

- 目标：构建可重复执行的前端验证基线。
- 具体事项：
  - 增加表达式优先级测试。
  - 增加嵌套 if/while 测试。
  - 增加全局声明 + 函数定义混合测试。
  - 增加返回值和嵌套块测试。

### 任务 5：准备与角色 B 的接口契约

- 目标：确保 AST 的结构定义与角色 B 的语义分析需求一致。
- 具体事项：
  - 明确 `Expr`、`Stmt`、`FuncDef`、`CompUnit` 的语义含义。
  - 对 `const`、`var`、`return`、`if/else`、`while` 的 AST 表示保持一致。
  - 记录关键字段的使用约定，避免后续接口不一致。

## 4. 交付物

- 完善后的 [src/lexer.cpp](../src/lexer.cpp)
- 完善后的 [src/parser.cpp](../src/parser.cpp)
- 追加的前端测试用例
- 可用于调试的 AST 输出工具

## 5. 验收标准

- 词法分析器能够稳定识别典型 ToyC 源码中的全部关键 token。
- 语法分析器能够正确构造 AST，并通过角色 A 自测。
- 角色 B 可以直接基于 AST 继续进行语义分析与 IR 生成。

## 6. 建议执行顺序

1. 先稳定词法分析器与测试。
2. 再收紧语法分析器覆盖面。
3. 最后补上 AST 调试输出和回归测试。
