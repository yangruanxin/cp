# ToyC 编译器 — 需求分析与任务分配

## 1. 项目概述

实现 ToyC 语言编译器，将 ToyC 源码编译为 RISC-V32 汇编。stdin 读入，stdout 输出，支持 `-opt` 参数。

**语言：** C++（C++20），构建系统：CMake

**时间：** 1 周 — 四人完全闭环、完全并行

## 2. 三阶段架构设计

核心思路：**引入线性 IR 作为中间表示，将编译器拆成三个完全解耦的阶段**。

```
源码字符串
    │
    ▼
┌──────────────────────────┐
│  A: Lexer + Parser        │  ← 只认文本和 AST
│  源字符串 → 完美 AST       │
└──────────┬───────────────┘
           │ AST (unique_ptr<CompUnit>)
           ▼
┌──────────────────────────┐
│  B: 语义分析 + IR 生成     │  ← 只认 AST 和指令数组
│  AST → 线性 IR (三地址码)  │
└──────────┬───────────────┘
           │ IR (vector<IRInstr>)
           │        ↕
           │  D: 优化器 (IR → IR)
           ▼
┌──────────────────────────┐
│  C: 代码生成              │  ← 只认指令数组和汇编
│  IR → RISC-V32 汇编       │
└──────────┬───────────────┘
           │ 汇编字符串
           ▼
        output.s
```

### 为什么能完全并行？

| 模块 | 输入 | 输出 | 自测方式 |
|------|------|------|----------|
| **A** | 源码字符串 | `CompUnit*` (AST) | 丢几段复杂 ToyC 代码，打印 AST 树 |
| **B** | `CompUnit*` (AST) | `vector<IRInstr>` | **手工 `new` AST 节点**喂进去，检查 IR 输出 |
| **C** | `vector<IRInstr>` | 汇编字符串 | **手写 IR 指令数组**喂进去，检查汇编输出 |
| **D** | `vector<IRInstr>` | 优化后 `vector<IRInstr>` | **伪造含死代码的 IR**，看优化器能否消除 |

**每个人都可以第 1 天开始写代码，第 5 天 D 负责把四段代码拼起来。**

## 3. 唯一需要 Day 1 敲定的契约

两个人形文件必须第 1 天一起敲定，之后**绝不修改**：

### 3.1 `src/ast.h` — A 和 B 的契约

A 定义 AST 节点结构（`Expr`、`Stmt`、`FuncDef`、`CompUnit`），B 确认可用。

### 3.2 `src/ir.h` — B 和 C 的契约（最关键！）

C 和 B 共同定义 IR 指令集，这是整个项目的核心枢纽：

```cpp
enum class IROp {
    FUNC_BEGIN, FUNC_END,    // 函数边界
    LABEL, GOTO, IF_GOTO,    // 控制流
    ADD, SUB, MUL, DIV, MOD, // 算术
    LT, GT, LE, GE, EQ, NE,  // 关系
    AND, OR,                 // 逻辑
    NEG, NOT,                // 一元
    LOAD_IMM,                // 加载立即数
    ASSIGN, DECL, LOAD, STORE, // 变量/内存
    CALL, RETURN, ARG        // 函数调用
};

struct IRInstr { IROp op; string dst, src1, src2; int intValue; string label; };
using IRList = vector<IRInstr>;
```

B 确保 AST 能正确翻译成 IR，C 确保 IR 能正确翻译成汇编。**C 不需要知道 if、while 是什么**——B 已经把它们降级成了 LABEL/GOTO/IF_GOTO。

## 4. 四人详细任务分配

---

### 👨‍💻 A — 前端闭环：源字符串 → 完美 AST

**世界观：** A 的世界里只有文本和树。不需要管汇编、IR、优化。

**核心任务：** 把 ToyC 源码变成一棵完整的抽象语法树（AST）。

**依据：** `任务要求.md` 中的 ToyC 文法是**完全确定**的，逐条翻译即可。

**交付物：**
| 文件 | 说明 |
|------|------|
| `src/lexer.h` / `lexer.cpp` | Token 识别、注释跳过 |
| `src/ast.h` | AST 节点定义（A 维护，全体确认） |
| `src/parser.h` / `parser.cpp` | 递归下降解析器 |
| `tests/test_lexer.cpp` | 自测：源字符串 → Token 序列 |
| `tests/test_parser.cpp` | 自测：Token 序列 → AST 打印 |

**接口：**
```cpp
Lexer(const string& src) → vector<Token>
Parser(const vector<Token>&) → unique_ptr<CompUnit>
```

**自测（不依赖任何人）：**
在 `test_parser.cpp` 中写：
```cpp
Lexer lexer("int main() { if (a > 0) return 1; else return 0; }");
auto tokens = lexer.tokenize();
Parser parser(tokens);
auto ast = parser.parse();
// 打印 AST 树，肉眼确认正确
```

**A 的输出就是 B 的输入。** A 完成后通知 B 和 D。

---

### 👩‍💻 B — 语言规范闭环：AST → 线性 IR

**世界观：** B 是"质检员"和"翻译官"。负责检查 AST 语义是否正确，然后把树状 AST **拍扁**成线性的三地址码（IR）。

**核心任务（两件）：**
1. **语义检查：** 符号表、作用域、类型、const、break/continue、return 路径
2. **IR 生成：** 用 Visitor 遍历 AST，把每个节点翻译成 1~N 条 IR 指令

**IR 生成对照表（示例）：**

| AST 节点 | 生成的 IR 序列 |
|----------|---------------|
| `1 + 2` | `LOAD_IMM t1, 1` → `LOAD_IMM t2, 2` → `ADD t0, t1, t2` |
| `a = b + 1` | `LOAD t1, a_fp, a_offset` → `LOAD_IMM t2, 1` → `ADD t0, t1, t2` → `STORE a_fp, a_offset, t0` |
| `if (c) S1 else S2` | `...计算 c→t0...` → `IF_GOTO t0, L1` → `...S1...` → `GOTO L2` → `LABEL L1` → `...S2...` → `LABEL L2` |
| `while (c) S` | `LABEL L1` → `...计算 c→t0...` → `IF_GOTO t0, L2` → `...S...` → `GOTO L1` → `LABEL L2` |
| `f(a, b)` | `ARG a, 0` → `ARG b, 1` → `CALL t0, f` |

**交付物：**
| 文件 | 说明 |
|------|------|
| `src/symbol_table.h` / `.cpp` | 嵌套作用域符号表 |
| `src/semantic.h` / `semantic.cpp` | 语义检查 + IR 生成（核心） |
| `src/ir.h` | IR 指令定义（B+C 共同确认） |
| `tests/test_semantic.cpp` | 自测：手工造 AST 节点，检查语义和 IR 输出 |

**接口：**
```cpp
class IRGenerator {
    IRList generate(CompUnit* ast);  // 语义检查 + IR 生成
};
```

**自测（不依赖 A）：**
```cpp
// 手工捏一个 AST: return 1+2;
auto ret = make_unique<Stmt>(); ret->kind = StmtKind::RETURN;
auto expr = make_unique<Expr>(); expr->kind = ExprKind::BINARY;
expr->binaryOp = BinaryOp::ADD;
expr->lhs = make_unique<Expr>(); expr->lhs->kind = ExprKind::INT_LITERAL; expr->lhs->intValue = 1;
expr->rhs = make_unique<Expr>(); expr->rhs->kind = ExprKind::INT_LITERAL; expr->rhs->intValue = 2;
ret->retExpr = move(expr);
// 喂给 IRGenerator → 验证 IR 输出是否包含 LOAD_IMM + ADD
```

### 👨‍💻 C — 后端闭环：线性 IR → RISC-V 汇编

**世界观：** C 根本不需要知道什么是 if、什么是 while。C 只认识 `vector<IRInstr>` 里的一条条指令，逐行翻译成 RISC-V 即可。

**核心任务：** 把扁平的 IR 指令数组翻译成 RISC-V32 汇编。

**IR → 汇编对照表：**

| IR 指令 | 生成的 RISC-V 汇编 |
|---------|-------------------|
| `LOAD_IMM t0, 42` | `li t0, 42` |
| `ADD dst, s1, s2` | `add dst, s1, s2` |
| `LABEL L1` | `L1:` |
| `GOTO L1` | `j L1` |
| `IF_GOTO cond, L1` | `bnez cond, L1` |
| `STORE base, off, val` | `sw val, off(base)` |
| `LOAD dst, base, off` | `lw dst, off(base)` |
| `CALL dst, fn` | `jal ra, fn` / `mv dst, a0` |
| `RETURN` | `mv a0, src` / ret |
| `FUNC_BEGIN fn` | `.globl fn` / `fn:` / prologue |
| `FUNC_END` | epilogue / `jalr zero, ra, 0` |

**寄存器分配策略（简单够用）：**
- 临时寄存器：`t0` ~ `t6` — 用于表达式求值
- 参数寄存器：`a0` ~ `a7` — 用于函数调用参数
- 返回值：`a0`
- 栈帧：`sp`、`s0`(fp)、`ra`

**交付物：**
| 文件 | 说明 |
|------|------|
| `src/codegen.h` / `codegen.cpp` | IR 到 RISC-V 汇编翻译器 |
| `tests/test_codegen.cpp` | 自测：手写 IR 数组 → 验证汇编 |

**接口：**
```cpp
class CodeGenerator {
    string generate(const IRList& ir);
};
```

**自测（不依赖 B）：**
```cpp
IRList ir = {
    IRInstr::funcBegin("main"),
    IRInstr::li("t0", 42),
    IRInstr::assign("a0", "t0"),
    IRInstr::ret(),
    IRInstr::funcEnd(),
};
CodeGenerator cg;
cout << cg.generate(ir);
// 期望输出: 包含 li t0, 42 和 mv a0, t0 的完整汇编
```

### 👩‍💻 D — 质量与性能闭环：工程 + 优化 + 集成 + 报告

**世界观：** D 是"总装车间主任"和"优化大师"。

**核心任务：**

| 优先级 | 任务 | 时间 |
|--------|------|------|
| P0 | `CMakeLists.txt` — 构建系统 | Day 1 |
| P0 | `main.cpp` — 串联 A→B→D(opt)→C | Day 3 |
| P1 | 优化器 — IR 常量传播 + 死代码消除 | Day 2-4 |
| P1 | 功能测试 `.tc` 文件 — 覆盖各语法特性 | Day 2-4 |
| P2 | 实践报告 | Day 6-7 |
| P3 | 集成联调 — 把 A+B+C 拼起来跑通 | Day 5-6 |

**优化器（D 的核心技术贡献）：**

```cpp
class Optimizer {
    IRList optimize(const IRList& input);  // 输入 IR，输出优化后 IR
};
```

优化 pass：
1. **常量传播：** 如果 `LOAD_IMM t0, X` 后跟 `ADD t1, t0, Y` 且 t0 仅被使用一次，直接计算 X+Y
2. **死代码消除：** 如果定义了临时变量但后续未使用，删除该指令
3. **常量折叠：** 编译期计算常量表达式

**集成 main.cpp：**
```cpp
int main(int argc, char* argv[]) {
    string src = read_stdin();
    auto tokens = Lexer(src).tokenize();     // A
    auto ast = Parser(tokens).parse();        // A
    IRList ir = IRGenerator().generate(ast);  // B
    if (optimize) ir = Optimizer().optimize(ir); // D
    cout << CodeGenerator().generate(ir);     // C
}
```

**交付物：**
| 文件 | 说明 |
|------|------|
| `CMakeLists.txt` | 构建系统 |
| `src/main.cpp` | 入口串联 |
| `src/optimizer.h` / `.cpp` | IR 级优化 |
| `tests/test_integration.cpp` | 端到端测试 |
| `tests/功能测试用例/*.tc` | ToyC 测试文件 |
| `docs/实践报告.md` | 报告 |

**自测：**
```cpp
IRList ir = {
    IRInstr::li("t0", 3),
    IRInstr::li("t1", 4),
    IRInstr::add("t2", "t0", "t1"),
    IRInstr::li("t3", 5),     // 死代码（t3 后续未用）
    IRInstr::assign("a0", "t2"),
    IRInstr::ret(),
};
Optimizer opt;
auto optimized = opt.optimize(ir);
// 验证: t3 的 li 被删除，或者 t0+t1 在编译期被折叠成 7
```

## 6. 如何整合为完整编译器

第 5 天，D 执行以下步骤：

```
Step 1: git pull A 的 ast.h/lexer/parser 分支
Step 2: git pull B 的 symbol_table/semantic 分支 + ir.h
Step 3: git pull C 的 codegen 分支
Step 4: git pull D 的 main.cpp/optimizer/CMakeLists.txt
Step 5: 在 main.cpp 中依次调用:
        Lexer → Parser → IRGenerator → (Optimizer) → CodeGenerator
Step 6: cmake --build . && ./compiler < test.tc > test.s
Step 7: 修复接口不匹配（字段名、类型签名等）
Step 8: 跑通全部测试用例
```

因为接口契约 `ast.h` 和 `ir.h` 是 Day 1 冻结的，**Step 5-7 通常只需要半天**。
