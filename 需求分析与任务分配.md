# ToyC 编译器 — 需求分析与任务分配

## 1. 项目概述

实现 ToyC 语言编译器，将 ToyC 源码编译为 RISC-V32 汇编。stdin 读入，stdout 输出，支持 `-opt` 参数。

**语言：** C++（C++20），构建系统：CMake

**时间：** 1 周 — 全员并行开发

## 2. 并行架构设计

核心思路：**先定接口，四路并行，最后集成**。

```
                    ┌─────────────────────────┐
                    │  D: 测试框架 + 测试用例   │
                    │  (手写 ToyC .tc 文件)    │
                    └──────────┬──────────────┘
                               │ 集成测试
    ┌──────────┐   ┌──────────▼──────────┐   ┌──────────┐
    │ A: Lexer │──▶│  main.cpp (D 编写)   │──▶│ C: CodeGen│
    │   +      │   │  串联 A+B+C 模块      │   │   + Opt  │
    │  Parser  │──▶│                      │──▶│          │
    └──────────┘   └──────────┬──────────┘   └──────────┘
                              │
                    ┌─────────▼─────────┐
                    │ B: SemanticAnalyzer│
                    └───────────────────┘
```

**每个人的独立测试策略**（确保互不阻塞）：

| 角色 | 测试方式 |
|------|----------|
| **A** | 直接喂字符串给 Lexer，检查 Token 序列；喂 Token 给 Parser，检查 AST 结构 |
| **B** | 在测试代码中**手工构造 AST 对象**，喂给 SemanticAnalyzer，验证报错/通过 |
| **C** | 在测试代码中**手工构造 AST 对象**，喂给 CodeGenerator，验证汇编输出是否正确 |
| **D** | 写完整的 `.tc` 源文件，集成后跑端到端测试 |

## 5. 接口约定（Day 1 必须敲定）

所有模块通过以下头文件耦合。Day 1 A/B/C/D 一起确认这些接口：

### 5.1 `src/ast.h` — AST 节点定义

```cpp
// 关键类型 enum class：ExprKind, UnaryOp, BinaryOp, StmtKind, FuncType
// 关键 struct：Expr, Stmt, Param, FuncDef, CompUnit
// 全部用 unique_ptr 管理所有权
```

### 5.2 模块 API 签名

| 模块 | 入口函数 | 所在文件 |
|------|----------|----------|
| Lexer | `Lexer(string) → vector<Token>` | `lexer.h` |
| Parser | `Parser(vector<Token>) → unique_ptr<CompUnit>` | `parser.h` |

| CodeGen | `CodeGenerator::generate(CompUnit*) → string` | `codegen.h` |
| Optimizer | `Optimizer::optimize(CompUnit*) → void` | `optimizer.h` |

## 7. 一周完成的核心策略

1. **先跑通，再优化** — Day 5 首要目标是 10 个基础功能测试通过，优化放到最后
2. **不做完美寄存器分配** — 先用临时寄存器 t0-t6，编译结果正确即可
3. **测试驱动** — 每人先写测试再写实现
4. **D 是救火队员** — 谁进度慢了 D 就支援谁
5. **每日站会（15 分钟）** — 同步进度、暴露阻塞
