# ToyC 编译器 — 需求分析与任务分配

## 1. 项目概述

实现 ToyC 语言的编译器，将 ToyC 源文件编译为 RISC-V32 汇编代码。编译器从 stdin 读取源码，向 stdout 输出汇编代码，支持 `-opt` 参数开启优化。

**语言选择：** C++（C++20）或 Java（JDK 21），建议用 C++。

## 2. 需求分析

### 2.1 总体架构

```
ToyC 源码 → [词法分析] → Token 流 → [语法分析] → AST → [语义分析] → 带标注的 AST → [代码生成] → RISC-V32 汇编
                                                                                    ↑
                                                                              [优化器]
```

### 2.2 模块划分

| 模块 | 子任务 | 说明 |
|------|--------|------|
| **词法分析** | 识别终结符 | ID、NUMBER、关键字(int/void/const/if/else/while/break/continue/return)、运算符、分隔符、(、)、{、}、;、, |
| | 跳过空白/注释 | 空格、制表符、换行；单行注释 //；多行注释 /* */ |
| | Token 封装 | 每个 Token 包含类型、值、位置信息（行号/列号） |
| **语法分析** | 递归下降 / ANTLR | 根据 ToyC 文法构建具体语法树或 AST |
| | 文法覆盖 | CompUnit、Decl、ConstDecl、VarDecl、Stmt、Block、FuncDef、Param、Expr 等全部产生式 |
| | 错误恢复 | 基本的语法错误提示 |
| **语义分析** | 符号表 | 全局/局部作用域嵌套，支持同名屏蔽 |
| | 类型检查 | int/void 返回值匹配、表达式类型正确性 |
| | 常量折叠 | const 常量编译期求值 |
| | 语义约束 | main 函数存在且签名为 int()；break/continue 只能在循环内；return 路径覆盖；函数声明先于调用；除数非零检测 |
| **代码生成** | 指令选择 | RISC-V32 指令集（add, sub, mul, div, rem, slt, and, or, xori, lw, sw, jal, jalr, beq, bne, blt, bge 等） |
| | 栈帧管理 | 函数调用约定，栈帧分配与恢复 |
| | 寄存器分配 | 简单分配或优化分配 |
| | 调用约定 | ra 保存、参数传递（a0-a7）、返回值（a0） |
| **优化器** | 常量传播 | \(选做\) |
| | 公共子表达式消除 | \(选做\) |
| | 死代码删除 | \(选做\) |
| | 寄存器分配优化 | \(选做\) |
| **构建与测试** | CMake 配置 | 构建系统配置 |
| | 本地测试 | 单元测试 + 集成测试 |
| | 性能评估 | 与 gcc -O2 对比 |

### 2.3 关键约束

- **输入/输出：** stdin → stdout，stderr 自由使用
- **入口点：** `int main(void)` 必须存在
- **初始化：** 所有变量/常量声明必须带有初始化表达式
- **常量：** const 初始化表达式只能含字面量、已有常量和它们的运算
- **短路求值：** && 和 || 必须实现短路逻辑
- **返回值：** int 函数所有路径必须有 return，void 函数不能有返回值

## 3. 四人任务分配

### 方案：按编译阶段拆分（推荐）

| 人员 | 角色 | 负责模块 | 核心交付物 | 预估工时 |
|------|------|----------|------------|----------|
| **A** | 前端工程师 | **词法分析 + 语法分析** | Lexer 实现、Parser 实现、AST 节点定义、语法错误报告 | ~5 天 |
| **B** | 语义工程师 | **语义分析 + 符号表** | 符号表实现、作用域管理、类型检查、常量折叠、语义约束检查 | ~4 天 |
| **C** | 后端工程师 | **代码生成 (RISC-V32)** | 指令选择、栈帧管理、函数调用约定、汇编输出、基础寄存器分配 | ~6 天 |
| **D** | 优化与集成 | **优化器 + 集成测试 + 实践报告** | 优化 pass 实现、构建系统配置、测试用例编写、实践报告撰写、整体联调 | ~5 天 |

### 3.1 详细职责

#### A — 词法分析 + 语法分析

1. 实现 `Lexer` 类：从字符流识别 Token
2. 实现 `Parser` 类（递归下降或 ANTLR 生成）：根据文法构造 AST
3. 定义 AST 节点（`CompUnit`、`FuncDef`、`Stmt`、`Expr` 等层级）
4. 提供语法错误报告（行号、期望与实际 Token）

#### B — 语义分析

1. 实现 `SymbolTable` 类：支持嵌套作用域、查找、插入
2. 遍历 AST 进行类型检查：
   - 函数返回值类型匹配
   - 赋值左右值类型匹配
   - 条件表达式必须为 int
   - 函数参数数量与类型匹配
3. 常量折叠：在编译期计算 const 表达式的值
4. 语义约束检查：
   - main 函数存在且签名正确
   - break/continue 在循环内
   - 函数必须先声明后调用
   - int 函数所有路径有 return
   - const 变量不能作为左值

#### C — 代码生成

1. 实现 `CodeGenerator` 类：遍历语义标注后的 AST 生成 RISC-V32 汇编
2. 指令选择：将 AST 节点映射到对应 RISC-V 指令
3. 栈帧管理：函数 prologue/epilogue（sp 调整、ra/s0 保存恢复）
4. 寄存器分配：实现基础分配（颜色分配或简单循环）
5. 短路求值：为 && 和 || 生成正确的分支跳转
6. 函数调用：参数传递（a0-a7）、jal 调用、返回值获取

#### D — 优化 + 集成 + 报告

1. 实现优化 pass（了解 C 组寄存器分配后可做更高级的优化）：
   - 常量传播与常量折叠
   - 死代码删除
   - 公共子表达式消除（选做）
2. 构建系统配置：`CMakeLists.txt`
3. 编写测试用例：覆盖各语法特性及边界情况
4. 集成联调：确保各模块对接正确
5. 编写实践报告（模板在项目文档中）

### 3.2 依赖关系与协作

```
A (Lexer) → A (Parser) → B (Semantic) → C (CodeGen) → D (Integration)
                                     ↕                ↕
                                  D (Optimizer)    D (Report)
```

- A 和 B 可以并行：A 完成后提供 Parser 接口给 B 使用
- B 和 C 可以部分并行：B 定义语义标注接口后 C 即开始编码
- D 全程参与：前期准备构建系统，中期写测试用例，后期集成优化 + 报告

### 3.3 通信与协作规范

1. **接口约定：** 各模块间接口（AST 节点定义、符号表 API 等）在编码开始前共同敲定
2. **代码审查：** 合入主分支前至少 1 人 review
3. **每日同步：** 简短同步进度与遇到的问题
4. **Git 分支策略：** 每人一个 feature 分支，完成模块后合并到 main
